Especificação Técnica — Substituto Moderno para flatpak-pip-generator
Nome provisório

pfmr

(Python Flatpak Manifest Resolver)

1. Objetivo

Criar uma ferramenta moderna para geração automática de módulos para manifesto Flatpak para dependencias Python, substituindo funcionalmente o flatpak-pip-generator, porém resolvendo corretamente:

dependências transitivas Python;
dependências nativas do sistema;
dependências providas pelo SDK Flatpak;
dependências providas por SDK Extensions;
dependências externas não empacotadas;
toolchains adicionais (Rust, Go, LLVM, Fortran, etc.);
wheels vs source distributions;
builds reprodutíveis;
manifests incrementalmente cacheáveis.

A ferramenta deve utilizar o ecossistema moderno Python (uv) como backend primário de resolução.

2. Problemas do flatpak-pip-generator
2.1 Limitações atuais
Problema	Impacto
resolução incompleta de dependências	builds quebram
ignora dependências nativas	erros de compilação
não entende SDKs Flatpak	recompilação desnecessária
não entende SDK Extensions	toolchains ausentes
não classifica wheels vs sdists	builds lentos
sem cache inteligente	rebuilds enormes
sem banco de conhecimento	repetição de hacks
sem resolução de bibliotecas externas	manutenção manual
sem lockfile robusto	baixa reprodutibilidade
3. Objetivos principais
3.1 Objetivos funcionais

A ferramenta deve:

Resolver dependências Python com uv
Detectar dependências nativas automaticamente
Identificar dependências já presentes:
no runtime;
no SDK;
em SDK Extensions.
Gerar módulos Flatpak otimizados
Resolver bibliotecas externas automaticamente
Aprender progressivamente com manifests conhecidos
Permitir bancos externos de resolução
Ser deterministicamente reproduzível
Minimizar recompilações
Produzir manifests legíveis e editáveis

5. Componentes
5.1 UV Resolver Engine
Objetivo

Substituir completamente o uso de pip.

Backend

Usar:

uv

Responsabilidades
resolver dependências transitivas;
gerar lockfile determinístico;
identificar wheels disponíveis;
detectar backends de build;
produzir árvore completa de resolução.

Entrada

      "version": "43.0.0",

  ]
}

17. Diferenciais principais

A nova ferramenta deve:

entender o ecossistema Flatpak;
entender toolchains;
entender runtimes;
entender SDK Extensions;
entender bibliotecas nativas;
aprender progressivamente;
produzir builds reproduzíveis;
minimizar manutenção manual.

Ela deve funcionar mais como um “gerenciador inteligente de empacotamento Flatpak” do que apenas um conversor de requirements Python.

18.1 Princípio fundamental

A ferramenta NÃO simula o ambiente Flatpak.

Ela sempre executa dentro de:

flatpak-builder
ou flatpak build
ou flatpak run

Ou seja:

O Flatpak runtime é o sandbox. O pfmr apenas orquestra.

18.2 Modos de execução de sandbox
Modo A — Execução direta no SDK (preferencial)

Usando:

flatpak-builder --run build-dir manifest.json bash
Uso:
testes de instalação Python (uv pip install)
testes de build
inspeção de runtime
Modo B — Execução controlada via flatpak build
flatpak build build-dir command...
Uso:
execução isolada sem rebuild completo
validação incremental
testes rápidos de dependência
Modo C — Execução de aplicação temporária
flatpak run --command=bash app.id
Uso:
validação pós-build
teste de runtime final
verificação de permissões
18.3 Criação da sandbox de teste

O pfmr deve gerar automaticamente um manifest de teste temporário, derivado do manifest principal.

Estrutura gerada
app-id: org.pfmr.TestSandbox
runtime: org.freedesktop.Platform
sdk: org.freedesktop.Sdk

modules:
  - name: pfmr-test-env
    buildsystem: simple
    build-commands:
      - python3 -m venv /app/venv
      - /app/venv/bin/pip install uv
Características
isolado do app principal
sem persistência
destruído após execução
pode ser regenerado a qualquer momento
18.4 Injeção de cenário de teste

O pfmr deve suportar cenários:

1. Teste de instalação Python
uv pip install package
2. Teste de importação
python -c "import numpy"
3. Teste de binários nativos
ldd $(python -c "import numpy; print(numpy.__file__)")
4. Teste de build backend
setuptools
maturin
meson-python
18.5 Captura de falhas

O sandbox deve capturar:

stderr completo
exit code
missing libraries
missing headers
missing pkg-config entries

Exemplo:

ERROR: libusb-1.0.so not found
Normalização de erro
{
  "error_type": "missing_native_dependency",
  "missing": "libusb-1.0",
  "source": "ldd",
  "context": "numpy build"
}
18.6 Integração com SDK e Extensions

A sandbox deve ser criada SEMPRE com:

SDK alvo
Runtime alvo
Extensions declaradas no manifest

Exemplo:

sdk-extensions:
  - org.freedesktop.Sdk.Extension.rust-stable
18.7 Determinismo da sandbox

Cada execução deve ser:

reproduzível
sem cache host
sem vazamento de sistema local

Regras:

PATH controlado
env vars fixas
sem acesso a /usr do host
apenas SDK + runtime
18.8 Cache de sandboxes

O pfmr pode cachear:

imagem do SDK base
layers de dependência Python
builds intermediários

Mas NUNCA:

resultados de falha
estado de sandbox quebrada
18.9 Filosofia de design

A sandbox não é um container genérico.

Ela é:

uma execução fiel do ambiente Flatpak alvo com instrumentação de erro.

18.10 Resultado final esperado

Após execução, o pfmr deve ser capaz de responder:

quais dependências Python faltam
quais libs nativas estão ausentes
se o SDK atual é suficiente
se SDK extension é necessária
se o build é possível sem modificações
